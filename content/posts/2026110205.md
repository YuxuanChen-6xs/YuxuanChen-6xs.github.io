## 代价最高的3个坏习惯（附Java代码避坑）

### 坏习惯一：每次调用都完整携带历史对话

这是Java开发者最容易踩的坑。

很多教程里给的示例代码是“一次问答，一次请求”。但在实际项目中，你总得支持多轮对话（比如客服机器人、文档对话）。于是很多人直接把整个 `messages` 数组不设上限地塞进请求体：

java
List<ChatMessage> messages = new ArrayList<>();
messages.add(new ChatMessage("system", "你是一个助手"));
messages.add(new ChatMessage("user", userInputHistory)); // 随着对话变长，这里会膨胀


**问题在哪里？**

Token是按字符计费的。你把所有历史回复都喂给AI看一遍，相当于每次对话都在花冤枉钱：不仅消耗输入Token，AI模型还要“看”一遍几千字的历史记录，才能生成几个字的回答。

**避坑Java代码示例：**

你应该设一个最多保留N轮的控制逻辑。比如只保留最近三轮对话：

java
public class TokenSaver {
    private static final int MAX_ROUNDS = 3;
    private List<ChatMessage> conversation = new LinkedList<>(); // 用链表方便开头删除

    public void addMessage(ChatMessage msg) {
        conversation.add(msg);
        if (conversation.size() / 2 > MAX_ROUNDS) { // 每轮包含user+assistant两条消息
            conversation.remove(0); // 删除最早的用户输入
            conversation.remove(1); // 删除最早的AI回复
        }
    }

    public List<ChatMessage> getContext() {
        return new ArrayList<>(conversation); // 只返回最近3轮
    }
}


别小看这个改动。如果你的用户平均对话轮次超过10轮，这个习惯每年能帮你省下**50%**以上的Token费用。

### 坏习惯三：同步阻塞调API，超时重试却无上限

Java 的 `RestTemplate` 或者 `HttpClient` 如果直接以同步方式调用 OpenAI，会阻塞当前线程。想象一下：一个 API 响应慢（例如2秒），你的线程池被打满，用户等急了，又发来几个请求——新的 HTTP 连接超时，你的重试机制开始重试3次。

每次重试，如果请求体里带着完整的对话历史（参考“坏习惯一”），那又是一笔不小的Token开销，而且最终结果还是没有返回。

更可怕的是：很多 Java 项目简单使用 `try-catch(Exception e)` 进行重试，失败后直接重试整个请求，这比直接输错结果还贵。

**避坑Java代码示例（带指数退避的重试）：**

java
import io.github.resilience4j.retry.Retry;
import io.github.resilience4j.retry.RetryConfig;
import java.time.Duration;

public class SmartRetry {
    RetryConfig config = RetryConfig.custom()
            .maxAttempts(3)
            .waitDuration(Duration.ofSeconds(1))
            .retryExceptions(IOException.class) // 只对网络或超时异常重试
            .build();

    Retry retry = Retry.of("openAICall", config);

    public String callAPI(Supplier<String> apiCall) {
        return Retry.decorateSupplier(retry, apiCall).get();
    }
}


这条规则明白着告诉JVM：如果OpenAI返回了 `429` 限流、或者 `Connection reset`，它只会以逐渐增大的间隔重试，而不是秒级重试3次。当服务端负载高时，你的请求量不会立马喷到 API 上去，避免了“重试风暴”的Token浪费。
