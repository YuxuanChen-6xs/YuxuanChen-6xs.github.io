## 错误一：没有合理限制Token输出长度

这应该是新手最严重的错误。调用Qwen API时，Java代码里如果不显式设置`max_tokens`，模型会按默认值输出。一些模型默认就拉满4096甚至8192个Token，一次调用可能花掉十几刀。

**正确做法：** 根据你的场景，把输出长度精准卡住。比如用千聚的API地址`https://www.qianjuai.com/v1`，代码里显式设置最大Token数。

**Java示例：**

java
import com.openai.client.OpenAiClient;
import com.openai.client.okhttp.OpenAiOkHttpClient;
import com.openai.models.*;

public class QwenCallExample {
    public static void main(String[] args) {
        OpenAiClient client = OpenAiOkHttpClient.builder()
                .baseUrl("https://www.qianjuai.com/v1")
                .apiKey("你的千聚API Key")
                .build();

        ChatCompletionCreateParams params = ChatCompletionCreateParams.builder()
                .model("qwen-plus")
                .maxCompletionTokens(500)  // 强制限制输出
                .addUserMessage("写一段300字的产品介绍")
                .build();

        client.chat().completions().create(params).stream()
                .forEach(choice -> System.out.println(choice.message().content().get()));
    }
}


你输出的长度缩减到500 Token以下，单次调用成本直接下降60%以上。

## 错误二：没有利用上下文缓存或系统消息精简

有些开发者每次请求都带上完整的历史对话记录，甚至连无关的系统字段都扔进去。Qwen API按Token计费，上下文越长，花费越高。

**正确做法：** 把重复的、固定的系统指令或提示词放在系统消息里，或者用千聚支持的缓存机制减少冗余Token。

**错误写法：**

// 每次请求都把全部历史对话拼进来，Token爆炸

**正确写法（Java示例）：**

ChatCompletionCreateParams params = ChatCompletionCreateParams.builder()

        .addUserMessage("今天有什么促销活动？")

把这个习惯改了之后，单次调用的Token开销减少近一半。

## 错误四：并发请求时没有设置重试和限流

有些Java代码一次性发送几百个并发请求，既没做重试守护，也没设置限流和退避。一旦API返回429（限流）或503（服务不可用），失败请求还是会消耗Token量（部分情况下）。

**正确做法：** 用Retry机制和指数退避。

**Java示例（使用开源库Retry4j）：**

java
import com.github.rholder.retry.*;

public class RetryUtil {
    public static Retryer<ChatCompletion> createRetryer() {
        return RetryerBuilder.<ChatCompletion>newBuilder()
                .retryIfExceptionOfType(RateLimitException.class)
                .withWaitStrategy(WaitStrategies.exponentialWait(1000, 30000, TimeUnit.MILLISECONDS))
                .withStopStrategy(StopStrategies.stopAfterAttempt(3))
                .build();
    }
}


这样不仅避免浪费重复调用的Token数，还能提高任务成功率。实测能把失败重试时的额外开销降低70%以上。
