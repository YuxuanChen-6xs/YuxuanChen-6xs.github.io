## 什么是异步调用，什么是同步调用

先打个比方。想象你去奶茶店点单。

**同步调用**：你站在柜台前，看着店员做奶茶，一步都不走开。店员做一杯，你就等一杯，什么其他事都不干，直到拿到奶茶。这期间，店里其他客人都在排队，队伍越来越长，但你就是不走。

**异步调用**：你点完单，扫码付了钱，拿到一个号牌，然后坐到旁边沙发上刷手机。店员做好奶茶后，叫号，你再去拿。这期间，你可以同时刷剧、回消息，甚至再点一杯别的。店里其他人也能同时被服务。

在Node.js里，这个道理一模一样。Node.js是单线程、非阻塞的事件驱动模型。**异步调用**允许你的程序在发起一个API请求后，先不干等着，而是继续执行下面的代码。等API响应回来了，通过回调函数、Promise或者async/await来通知你。**同步调用**则相反，程序会卡在当前这行代码上，直到API完全返回数据才会继续往下跑。

对于ERNIE API这种网络I/O密集型任务（一个请求从发出到返回，通常要花几百毫秒到几秒），用同步调用的后果就是CPU在那傻等，程序完全不干事。用异步调用，CPU可以在这段时间里去处理其他用户的请求，或者继续发起更多API调用。

## 为什么选错方案每月会多花300块

你可能会想：不就是个调用方式，至于差这么多钱吗？答案是，非常至于。

大多数云服务器是按**运行时间**和**并发连接数**收费的，或者你买的是固定配置的实例。如果你用同步方式，一个请求就独占一个进程，服务器处理10个用户就需要同时开10个进程，资源占用极高。

举个具体的场景：你写了一个AI聊天助手，用ERNIE API生成回复。如果每个用户请求都同步阻塞等待，你的服务器为了维持响应，就必须开大量进程或线程。假设你的服务器是4核8G的实例，月租大概200元左右。用同步模式，它可能只能稳定支持50个并发用户。但如果你改成异步模式，同样这台服务器，靠着Node.js单线程的事件轮询，轻松支撑200个并发用户，甚至更多。

这意味着什么？意味着当你用户量从50涨到200时，同步方案必须买更贵的服务器（比如升级到8核16G，月租500多），而异步方案一分钱不用多花。算一下：500 - 200 = 300。**每月省下300块**就是这么来的。

而且这还只是服务器钱。如果API是按调用量付费（[千聚api聚合站](https://www.qianjuai.com/)1元等于1美元Token额度），同步方案因为每个请求处理慢，连接超时、重试的次数更多，浪费的Token也更多，等于间接增加了Token消耗费用。异步方案因为响应及时、连接稳定，重试率低，Token利用率更高。

### 示例1：同步调用（不推荐，仅做对比）
javascript
const axios = require('axios');

// 同步调用函数（使用async/await包装，但内部用同步写法模拟阻塞）
async function syncCall(model, messages) {
    const url = 'https://www.qianjuai.com/v1/chat/completions';
    const headers = {
        'Authorization': `Bearer 你的千聚API_KEY`,
        'Content-Type': 'application/json'
    };
    const data = {
        model: model,
        messages: messages,
        stream: false
    };

    // 注意：这里虽然是await，但在单线程里它会阻塞当前“任务”，
    // 如果所有请求都走这个逻辑，相当于同步等待。
    try {
        const response = await axios.post(url, data, { headers });
        return response.data;
    } catch (error) {
        console.error('同步请求失败:', error.message);
        throw error;
    }
}

// 模拟连续调用多个请求（顺序执行）
async function main() {
    const start = Date.now();

    try {
        const res1 = await syncCall('ernie-3.5', [{ role: 'user', content: '你好，你是谁？' }]);
        console.log('第一次回答:', res1.choices[0].message.content);

        const res2 = await syncCall('ernie-3.5', [{ role: 'user', content: '今天天气好吗？' }]);
        console.log('第二次回答:', res2.choices[0].message.content);

        const res3 = await syncCall('ernie-3.5', [{ role: 'user', content: '推荐一本好书' }]);
        console.log('第三次回答:', res3.choices[0].message.content);
    } catch (error) {
        console.error('总出错:', error);
    }

    const end = Date.now();
    console.log(`同步模式总耗时: ${end - start}ms`);
}

main();
// 输出示例：假设每个请求平均2秒，总耗时约6秒


**这段代码的问题**：每次调用都等前一个完全结束才发起下一个，浪费了大量等待时间。如果用户量上来，这样的代码会导致服务器进程堆积，响应变得极慢。

### 示例2：异步调用（推荐方案）
javascript
const axios = require('axios');

// 异步调用函数（真正的非阻塞，返回Promise）
async function asyncCall(model, messages) {
    const url = 'https://www.qianjuai.com/v1/chat/completions';
    const headers = {
        'Authorization': `Bearer 你的千聚API_KEY`,
        'Content-Type': 'application/json'
    };
    const data = {
        model: model,
        messages: messages,
        stream: false
    };

    // 这里还是await，但因为是Promise.all并发，事件循环会同时调度多个请求
    try {
        const response = await axios.post(url, data, { headers });
        return response.data;
    } catch (error) {
        console.error('异步请求失败:', error.message);
        throw error;
    }
}

// 核心：并发发起多个请求，而不是顺序等待
async function main() {
    const start = Date.now();

    // 一次性创建三个任务，让它们同时执行
    const task1 = asyncCall('ernie-3.5', [{ role: 'user', content: '你好，你是谁？' }]);
    const task2 = asyncCall('ernie-3.5', [{ role: 'user', content: '今天天气好吗？' }]);
    const task3 = asyncCall('ernie-3.5', [{ role: 'user', content: '推荐一本好书' }]);

    // 等待所有任务完成（不会阻塞事件循环）
    try {
        const results = await Promise.all([task1, task2, task3]);
        results.forEach((result, index) => {
            console.log(`第${index + 1}次回答:`, result.choices[0].message.content);
        });
    } catch (error) {
        console.error('某个请求失败:', error);
    }

    const end = Date.now();
    console.log(`异步模式总耗时: ${end - start}ms`);
}

main();
// 输出示例：三个请求并发执行，总耗时约2秒（而不是6秒）


**这段代码的优势**：三个请求几乎同时发出，事件轮询让Node.js边等待第一个请求的响应，边处理另一个请求。总耗时从6秒降到2秒。如果请求数越多，这种优势越明显，并发量越高，你花的服务器钱越值。

## 两种调用方式的核心对比

| 对比维度         | 同步调用（顺序执行）                     | 异步调用（并发）                     |
| ------------------ | ------------------------------------------ | -------------------------------------- |
| 代码可读性       | 直观，符合习惯                           | 需要理解Promise.all等模式             |
| 响应速度         | 慢，总时间 = 所有请求时间之和               | 快，总时间 ≈ 最慢的那个请求时间         |
| 服务器资源占用   | 高，需要大量进程/线程                     | 低，单线程高效调度                     |
| 可扩展性         | 差，并发用户数一多就扛不住               | 强，轻松支撑高并发                     |
| Token浪费率      | 高，重试多                               | 低，连接稳定                           |
| 开发调试难度     | 简单，bug少                              | 略复杂，需注意Promise错误处理         |
| 适合场景         | 一次性、低频率的个体测试                 | 生产环境、高并发API服务               |

对于新手开发者，我的建议是：**只要是在真实项目里跑，无条件选择异步模式**。只有在你刚开始调试代码、确认API密钥是否正确这种“一次跑一个请求”的情况下，可以用同步模式简化步骤。一旦进入服务端开发阶段，立即切换为异步。
